# ゆかナビ マイページ連携仕様

アプリのマイページ (うたった曲 / あとで歌う / お気に入り曲 / お気に入り検索) と、
サーバー (Web 版マイページ)・Google Drive の連携仕様。実装は
`Core/MypageService.cs` / `Core/GoogleAccount.cs` / `Core/GoogleDrive.cs` (アプリ) と
`/api/mypage.php` (サーバー、API 仕様は KaraokeRequestorWeb の `api/README.md`) 。

## 設計思想

マイページのデータは本来**ユーザーの手元 (端末) にあるべきもの**。
ブラウザではローカル保存の仕組みが作れないため、Web 版はやむを得ずサーバー側に
データを持ち、その代替 + 部屋 (サーバー) をまたぐ持ち運び手段として Google Drive
保存がある。アプリは端末ローカル (`persistentDataPath/mypage.json`) が一次データで、
Google Drive はユーザー自身の保存場所 (アプリが直接読み書きする)。

## 3つの状態

| 状態 | 読み | 書き |
|---|---|---|
| **未リンク** (既定) | 端末ローカル | 端末ローカル |
| **リンク済み** (現在のサーバーとデバイスリンク済み) | サーバー (Web 版と同一) | サーバー + ローカルの二重書き |
| 別のサーバーに接続中 (そのサーバーとは未リンク) | 端末ローカル | 端末ローカル |

- リンクは**サーバー (部屋) ごと**に記憶される (`AppConfig.LinkedMypageUserId`)。
  リンク済みの部屋に接続し直すと自動的にサーバーモードへ戻る (再リンク不要)
- リンク済み中の読み込みで、お気に入り系 (お気に入り曲 / あとで歌う / お気に入り検索)
  はローカルへ**ミラー** (サーバー内容で置き換え) される。Web 側での追加・削除が
  ローカルにも反映され、別サーバーへ移ったときにそのまま使える
- **履歴はミラーしない**。ローカル履歴は端末の蓄積として保ち続ける
  (サーバーごとの履歴と端末の履歴は別物)。リンク中の予約はサーバー側 (exec.php) が
  自動でサーバー履歴に記録し、アプリはローカルにも記録する

## デバイスリンク

- Web 版の マイページ > デバイスリンク で発行した6文字コード (5分有効) を、
  アプリの マイページ > 連携 で入力する
- リンク時、端末内データを**一括統合** (import): 履歴は歌った回数ぶんの行に展開
  (日時は最終予約日時から1秒ずつずらす)、お気に入り系はそのまま。import は冪等
  (重複はサーバー側でスキップ) なので何度実行しても安全
- 統合後、サーバー内容をローカルへミラーして双方が揃う

### 別サーバーを経由して戻ったとき (自動再統合)

「最後に統合したサーバー」を端末に記録し、別のサーバーを経由してリンク済みサーバーへ
戻った最初の読み込み時に、端末データを自動で再統合する (別サーバー滞在中にローカルへ
貯めた履歴・お気に入りが反映される)。

### 制約: 削除の同期は片方向

削除は「行が無い」ことでしか表現されない (削除記録 = トゥームストーンを持たない) ため、
**別サーバー滞在中にローカルで削除したものは、リンク済みサーバーに戻ると復活しうる**。
複雑化回避のため許容する (Web 版のインポートも同じ挙動)。

### 連携解除

- 解除しても端末内データはそのまま残る
- 再連携はペアコード入力で (Google 同期はデバイスリンクと独立で、解除の影響を受けない)

## Google 同期 (アプリ直接の認証 + Drive 自動同期)

アプリが自分で Google にログインし、Drive (appDataFolder) の `mypage_data.json` を
直接読み書きする。**部屋 (サーバー) と無関係**にどこでも同じデータを使える。
部屋側に Google 設定 (client_id 等) は不要。

### 認証フロー (relay の app_auth)

1. 連携画面の「Google にログイン」 → アプリが 128bit 乱数のセッション ID を生成し、
   relay の `?action=app_auth&session=` をブラウザで開く
2. relay (既定 `https://ykr.moe/mypage_google_callback.php`) が Google OAuth へ誘導。
   Client ID / Secret は relay だけが持つ (アプリには URL 以外の秘密が無い)
3. 認証完了で relay がトークンを一時保存 (10分・ワンタイム) し、アプリが
   `?action=app_poll` のポーリング (2秒間隔・最大5分) で受け取って端末 (PlayerPrefs) に保存
4. 以降のアクセストークン更新は `?action=app_refresh` (refresh_token のみで更新)

中継サーバー URL は 設定画面 > 高度な設定 で変更できる (自前 relay を立てる場合のみ)。

### 同期タイミング (自動)

- **起動時**: ログイン済みなら Drive から取り込む (pull)。失敗は静かにスキップ
- **ローカル変更時**: 10秒の debounce で Drive へ保存 (push)。
  予約 (履歴追加)・お気に入り操作などすべてのローカル書き込みが対象
- push は「そのセッションで一度 Drive を読んだ後」だけ行う
  (取り込み前のローカルデータで Drive 側を上書きして消さないため)
- 手動同期: 連携画面の「Driveへ保存」「Driveから取込」

### 統合 (マージ) のルール

pull は置き換えではなく統合: 履歴 = 同じ曲は回数・最終日時の大きい方 /
あとで歌う・お気に入り曲 = fullpath の重複回避で追加 /
お気に入り検索 = 同条件の重複回避で追加 (Web 独自形式は `ServerType` を保持)。

### Web 版との互換

Drive 上のファイル・形式 (version 1) は Web 版の Google 連携と完全に同一。
Web 版で連携した部屋の自動同期とも同じデータを共有する
(部屋側の書き込みタイミングは Web 版仕様のまま = マイページ書き込み操作時)。

### トークンの失効

refresh_token が無効になるとログアウト扱いになり「Google の認証が切れました。
もう一度ログインしてください」と表示する。無効になる条件 (Google の仕様):

- ユーザーが Google アカウント設定でアクセス権を取り消した
- 6ヶ月間未使用
- **relay の OAuth 同意画面が「テスト」公開ステータスだと7日で失効** (本番公開が必要)

アクセストークン (約1時間) の失効は自動更新されるため、失効には当たらない。

### 連携データの削除 (アプリ内)

マイページ > 連携 の「連携データを削除」(2度押し確認) で以下を順に行う
(App Store 審査ガイドライン 4 のデータ削除要件に対応。`MypageService.DeleteDriveDataAsync`):

1. Drive 上の `mypage_data.json` を削除 (`GoogleDrive.DeleteAsync`。失敗したら中断し、再試行できる)
2. `https://oauth2.googleapis.com/revoke` にトークンを POST して grant を失効
   (`GoogleAccount.RevokeAndLogoutAsync`。ベストエフォート — 失敗しても次へ進む)
3. ローカルのトークンと最終同期時刻を消してログアウト

端末内のマイページデータは残る。revoke は同一 OAuth クライアントの grant 全体を
失効させるため、同じ Google アカウントの Web 版連携も再ログインが必要になりうる。

### 認証ブラウザ (iOS はアプリ内シート)

認証ページは Android / Windows では既定ブラウザ、iOS では SFSafariViewController の
アプリ内シートで開く (`InAppBrowser.cs` + `Assets/Plugins/iOS/YukaNaviSafariView.mm`。
App Store 審査ガイドライン 4 の「外部ブラウザでのサインイン不可」対応)。
ポーリング完了・中止・タイムアウトでシートは自動的に閉じ、ユーザーがシートを
手動で閉じた場合は約5秒の猶予 (完了直後の取りこぼし防止) 後に中止扱いになる。

### セキュリティ

- トークンは端末 (PlayerPrefs) にだけ保存される (部屋のサーバーには渡さない。
  サーバー API にもトークンを返すアクションは無い)
- app_poll の受け取りはセッション乱数 (128bit・10分・ワンタイム) が資格、
  app_refresh は refresh_token 自体が資格。アプリ⇔relay 間に共有シークレットは無い
- 既知の残存リスク: 攻撃者が自分のセッションで作った認証 URL を踏ませる攻撃
  (OAuth デバイスフローと同型)。relay の認証完了画面に注意文言を表示して緩和

## お気に入り検索の形式変換

アプリの保存検索 (lister あいまい / everything / exact) と Web 版の
favorite_keyword (`search_type` + `search_params`) は形式が異なる。

- アプリ → サーバー: everything → `search`、lister → `listerdb_songlist` +
  `param=anyword`、exact → `listerdb_songlist` + `param=<カラム名>&match=exact`
- サーバー → アプリ: 上記の逆。**Web 独自の形式 (`param=song_name` 等) は
  あいまい検索に丸めて表示するが、元の `search_type`/`search_params` を
  `ServerType`/`ServerParams` として保持し、サーバーへ送り返すときはそのまま使う**
  (丸め直すと別条件として重複登録されるため)
