# ゆかナビ マイページ連携仕様

アプリのマイページ (うたった曲 / あとで歌う / お気に入り曲 / お気に入り検索) と、
サーバー (Web 版マイページ)・Google Drive の連携仕様。実装は
`Core/MypageService.cs` (アプリ) と `/api/mypage.php` (サーバー、API 仕様は
KaraokeRequestorWeb の `api/README.md`) 。

## 設計思想

マイページのデータは本来**ユーザーの手元 (端末) にあるべきもの**。
ブラウザではローカル保存の仕組みが作れないため、Web 版はやむを得ずサーバー側に
データを持ち、その代替 + 部屋 (サーバー) をまたぐ持ち運び手段として Google Drive
保存がある。アプリは端末ローカル (`persistentDataPath/mypage.json`) が一次データ。

## 3つの状態

| 状態 | 読み | 書き |
|---|---|---|
| **未リンク** (既定) | 端末ローカル | 端末ローカル |
| **リンク済み** (現在のサーバーとデバイスリンク済み) | サーバー (Web 版と同一) | サーバー + ローカルの二重書き |
| 別のサーバーに接続中 (そのサーバーとは未リンク) | 端末ローカル | 端末ローカル |

- リンクは**サーバー (部屋) ごと**に記憶される (`AppConfig.LinkedMypageUserId`)。
  リンク済みの部屋に接続し直すと自動的にサーバーモードへ戻る (再リンク不要)
- リンク済み中の読み込みで、お気に入り系 (お気に入り曲 / あとで歌う / お気に入り検索)
  はローカルへ**ミラー** (サーバー内容で置き換え) される。Web 側での追加・削除が
  ローカルにも反映され、別サーバーへ移ったときにそのまま使える
- **履歴はミラーしない**。ローカル履歴は端末の蓄積として保ち続ける
  (サーバーごとの履歴と端末の履歴は別物)。リンク中の予約はサーバー側 (exec.php) が
  自動でサーバー履歴に記録し、アプリはローカルにも記録する

## デバイスリンク

- Web 版の マイページ > デバイスリンク で発行した6文字コード (5分有効) を、
  アプリの マイページ > 連携 で入力する
- リンク時、端末内データを**一括統合** (import): 履歴は歌った回数ぶんの行に展開
  (日時は最終予約日時から1秒ずつずらす)、お気に入り系はそのまま。import は冪等
  (重複はサーバー側でスキップ) なので何度実行しても安全
- 統合後、サーバー内容をローカルへミラーして双方が揃う

### 別サーバーを経由して戻ったとき (自動再統合)

「最後に統合したサーバー」を端末に記録し、別のサーバーを経由してリンク済みサーバーへ
戻った最初の読み込み時に、端末データを自動で再統合する (別サーバー滞在中にローカルへ
貯めた履歴・お気に入りが反映される)。

### 制約: 削除の同期は片方向

削除は「行が無い」ことでしか表現されない (削除記録 = トゥームストーンを持たない) ため、
**別サーバー滞在中にローカルで削除したものは、リンク済みサーバーに戻ると復活しうる**。
複雑化回避のため許容する (Web 版のインポートも同じ挙動)。

### 連携解除

- 解除しても端末内データはそのまま残る
- 解除したサーバーでは **Google 持ち歩きの自動リンクも停止**する (勝手に復活しない)。
  再連携はペアコード入力か「Google 同期で連携する」ボタンの明示操作で

## Google 同期

- **認証はブラウザで行う** (アプリの連携画面 > 「連携ページを開く」 → Web 版の
  relay 経由 OAuth)。アプリ内に OAuth 実装は持たない
- **自動 Drive 同期のトリガーはマイページの書き込み操作時** (お気に入り追加等)。
  予約時ではない (予約処理を遅くしないため。Web 版と同じ)。
  そのため「予約しただけ」の履歴が Drive に載るのは、次の書き込み操作か手動同期時
- 手動同期: 連携画面の「Driveへ保存」「Driveから取込」

## Google 同期の持ち歩き (部屋またぎ自動引き継ぎ)

一度設定すれば、部屋を移動するだけで自分のマイページが自動で付いてくる機能。

### 開始条件 (最初の1回だけ必要)

1. その部屋とデバイスリンク済み
2. その部屋で Google 連携済み (ブラウザ認証)
3. 連携画面で「**同期の持ち歩きを開始**」を押す (トークン一式を端末に保存)

### 新しい部屋での自動連携

未リンクの部屋で**マイページを開いたとき**、持ち歩きトークンがあれば自動で:
サーバーに Google アカウントのユーザーを用意 (既存があれば再利用) → Drive から復元 →
自動デバイスリンク → 端末データの統合、まで行う。
トースト「Google アカウントで同期を開始しました」が出る。

### 自動連携がスキップされる条件 (エラーは出さない)

| 条件 | 挙動 |
|---|---|
| その部屋の Google 同期が未設定 (client_id / relay_secret なし) | スキップ (ローカルモード)。サーバーが新 API 未対応の場合も同様 |
| その部屋のクライアント ID が持ち歩きトークンの発行元と違う | スキップ (トークンは破棄しない。手動連携時は「別の Google 連携設定」と表示) |
| その部屋で明示的に連携解除した | スキップ (「Google 同期で連携する」ボタンで解除を取り消せる) |

### トークンの失効

refresh_token が無効になると持ち歩きを破棄し「Google の認証が切れています。
連携をやり直してください」と表示する。無効になる条件 (Google の仕様):

- ユーザーが Google アカウント設定でアクセス権を取り消した
- 6ヶ月間未使用
- **relay の OAuth 同意画面が「テスト」公開ステータスだと7日で失効** (本番公開が必要)

アクセストークン (約1時間) の失効は自動更新されるため、失効には当たらない。

### セキュリティ

持ち歩きトークンは端末 (PlayerPrefs) に保存される。Web 版も各部屋のサーバー DB に
トークンを平文保存している設計なので同水準。持ち歩きは連携画面からいつでも
停止できる (端末からトークンを消す)。

## お気に入り検索の形式変換

アプリの保存検索 (lister あいまい / everything / exact) と Web 版の
favorite_keyword (`search_type` + `search_params`) は形式が異なる。

- アプリ → サーバー: everything → `search`、lister → `listerdb_songlist` +
  `param=anyword`、exact → `listerdb_songlist` + `param=<カラム名>&match=exact`
- サーバー → アプリ: 上記の逆。**Web 独自の形式 (`param=song_name` 等) は
  あいまい検索に丸めて表示するが、元の `search_type`/`search_params` を
  `ServerType`/`ServerParams` として保持し、サーバーへ送り返すときはそのまま使う**
  (丸め直すと別条件として重複登録されるため)
